# AGENTS.md - Tomatobot

## Project Overview

Go 1.22 Telegram bot application. Module path: `github.com/tomato3017/tomatobot`.
Uses Cobra CLI, Telegram Bot API, bun ORM (SQLite/PostgreSQL), zerolog logging,
and a plugin-style module system with a notification pub/sub bus.

## Build / Lint / Test Commands

```bash
# Install dependencies
make dep                # go mod download && go mod tidy

# Generate code (VERSION.txt, mocks, stringer)
make generate           # go generate version_gen.go && go generate ./...

# Build binary
make build              # deps + generate + CGO_ENABLED=0 GOOS=linux go build -o ./bin/ .

# Lint (requires Docker)
make lint               # golangci-lint v1.59.1 via Docker, default config

# Run all tests
make test               # go test -race -v ./...

# Run a single test function
go test -race -v -run TestPoller_publishWeatherForLocation ./pkg/modules/weather/

# Run a single suite test method
go test -race -v -run TestRunWCmdAddSuite/Test_WCmdAdd_addWeatherLocation ./pkg/modules/weather/

# Run all tests in a package
go test -race -v ./pkg/notifications/

# Build Docker image
make image              # docker build . --tag tomatobot:latest
```

Always run tests with `-race` flag as the Makefile does.

## Code Style Guidelines

### Formatting

- Standard `gofmt` formatting. No custom formatter config.
- Linting via `golangci-lint` with default rules (no `.golangci.yml` in repo).

### Imports

Imports use a **single block** with stdlib, third-party, and internal packages mixed
together (sorted alphabetically by the Go tooling). Do NOT manually add blank-line
separations between import groups -- follow the existing single-block style:

```go
import (
    "context"
    "fmt"
    tgbotapi "github.com/go-telegram-bot-api/telegram-bot-api/v5"
    "github.com/rs/zerolog"
    "github.com/tomato3017/tomatobot/pkg/bot/models"
    "regexp"
    "strings"
)
```

**Standard import aliases used in this codebase:**
- `tgbotapi` for `github.com/go-telegram-bot-api/telegram-bot-api/v5`
- `dbmodels` for `github.com/tomato3017/tomatobot/pkg/bot/models/db`
- `cmdmdls` for `github.com/tomato3017/tomatobot/pkg/command/models`

### Naming Conventions

- **Files:** lowercase, no underscores for word separation: `tomatobot.go`, `chatlogger.go`, `dbconn.go`
  - Command handler files use `cmd` prefix: `cmdwadd.go`, `cmdwremove.go`, `cmdwlist.go`
  - Interface definition files are named `iface.go`
  - Mock files: `{interfacename_lowercase}_mock.go` (generated by mockery)
- **Packages:** short, lowercase, single-word: `bot`, `config`, `db`, `modules`, `notifications`, `util`, `proxy`, `owm`
- **Exported types:** PascalCase: `Tomatobot`, `WeatherModule`, `NotificationPublisher`
- **Unexported types/functions:** camelCase: `sudoer`, `poller`, `newPoller`, `executeBot`
- **Constants:** SCREAMING_SNAKE_CASE for package-level constants (`OWMAPIHOST`, `OWMAPIONECALLPATH`), camelCase for scoped constants (`topicFmtWatch`)
- **Interfaces:** PascalCase, descriptive names: `BotModule`, `Publisher`, `TGBotImplementation`, `ChatLogger`
- **Enums:** iota-based int types with `go:generate stringer` (`DBType`, `TextData`)

### Error Handling

1. **Always wrap errors with context** using `fmt.Errorf`:
   ```go
   return fmt.Errorf("failed to load configuration file: %w", err)
   ```

2. **Sentinel errors** with `errors.New()` when needed:
   ```go
   var ErrSubExists = errors.New("subscription already exists")
   ```

3. **Check errors** with `errors.Is()`:
   ```go
   if errors.Is(err, context.Canceled) { ... }
   if errors.Is(err, sql.ErrNoRows) { ... }
   ```

4. **Log non-fatal errors** with zerolog:
   ```go
   logger.Error().Err(err).Msg("Failed to send heartbeat")
   ```

5. **Use `util.CloseSafely()`** for deferred cleanup of `io.Closer`:
   ```go
   defer util.CloseSafely(t.dbConn)
   defer util.CloseSafely(res.Body)
   ```

6. Reserve `logger.Fatal()` and `log.Fatalf()` for truly unrecoverable init failures only.

### Logging

Use `zerolog` structured logging everywhere. Obtain a logger from context or struct fields:
```go
logger := zerolog.Ctx(ctx)
logger.Info().Str("module", name).Msg("Module initialized")
logger.Error().Err(err).Msg("failed to handle bus message")
```

### Testing

- **Test framework:** standard `testing` package + `github.com/stretchr/testify` (`require`, `suite`, `mock`)
- **HTTP mocking:** `github.com/h2non/gock` for external API tests
- **Interface mocks:** generated by `mockery` (config in `.mockery.yaml`). Run `make generate` to regenerate.
- **Database tests:** use in-memory SQLite (`file::memory:?cache=shared`) with programmatic migrations via `sqlmigrate.MigrateDbSchema()`
- **Suite-based tests:** embed `suite.Suite`, use `SetupTest()`/`TearDownTest()` lifecycle, launch with:
  ```go
  func Test_RunXxxSuite(t *testing.T) {
      suite.Run(t, new(XxxSuite))
  }
  ```
- **Assertions:** use `require` (not `assert`) for fail-fast behavior:
  ```go
  require.NoError(t.T(), err)
  require.Equal(t.T(), expected, actual)
  ```

### Architecture Patterns

- **Module system:** each module implements `BotModule` interface (Initialize/Start/Shutdown) and registers commands during initialization. Modules listed in `getModuleRegistry()`.
- **Command system:** hierarchical with `BaseCommand` for subcommand routing, `SimpleCommand` for callbacks, middleware chains (`WithNArgs`, `WithAdminPermission`).
- **Functional options:** used for configuring `NotificationPublisher` and bot proxy.
- **Concurrency:** `github.com/oklog/run` for goroutine lifecycle management (run groups).
- **Configuration:** layered YAML + env var overrides (`envconfig`), strict YAML parsing (unknown fields cause errors), struct validation with `go-playground/validator`.

### Mock Generation

Mockery config is in `.mockery.yaml`. Mocks are generated in-package with expecter methods.
To regenerate all mocks: `make generate` (or `go generate ./...`).
The `mock_generator.go` file at the repo root contains the top-level `go:generate` directive.

### Search Commands

Always use `rg` (ripgrep) instead of `grep` when searching file contents.

## üîÑ Standard Workflow

**Clavix follows this progression:**

```
PRD Creation ‚Üí Task Planning ‚Üí Implementation ‚Üí Archive
```

**Detailed steps:**

1. **Planning Phase**
   - Run: `/clavix:prd` or `/clavix:start` ‚Üí `/clavix:summarize`
   - Output: `.clavix/outputs/{project}/full-prd.md` + `quick-prd.md`
   - Mode: PLANNING

2. **Task Preparation**
   - Run: `/clavix:plan` transforms PRD into curated task list
   - Output: `.clavix/outputs/{project}/tasks.md`
   - Mode: PLANNING (Pre-Implementation)

3. **Implementation Phase**
   - Run: `/clavix:implement`
   - Agent executes tasks systematically
   - Mode: IMPLEMENTATION
   - Agent edits tasks.md directly to mark progress (`- [ ]` ‚Üí `- [x]`)

4. **Completion**
   - Run: `/clavix:archive`
   - Archives completed work
   - Mode: Management

**Key principle:** Planning workflows create documents. Implementation workflows write code.

## üí° Best Practices for Generic Agents

1. **Always reference instruction files** - Don't recreate workflow steps inline, point to `.clavix/instructions/workflows/`

2. **Respect mode boundaries** - Planning mode = no code, Implementation mode = write code

3. **Use checkpoints** - Follow the CHECKPOINT pattern from instruction files to track progress

4. **Create files explicitly** - Use Write tool for every file, verify with ls, never skip file creation

5. **Ask when unclear** - If mode is ambiguous, ask: "Should I implement or continue planning?"

6. **Track complexity** - Use conversational mode for complex requirements (15+ exchanges, 5+ features, 3+ topics)

7. **Label improvements** - When optimizing prompts, mark changes with [ADDED], [CLARIFIED], [STRUCTURED], [EXPANDED], [SCOPED]
